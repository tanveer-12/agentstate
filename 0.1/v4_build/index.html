    How to use this document

    Every week has the same structure: - Files to work in — exactly which files, in order - What to write — specific instructions, not code - Docs to write — what user-facing documentation ships with each version - How to test — exact commands to run in PowerShell - How to know it worked — what passing looks like

    Three rules: 1. Run test commands at the end of every session, not just end of week 2. Docs ship with every version — users should be able to use what you publish immediately 3. Colab notebook stays running whenever you need local models

    Documentation Strategy — Read Before Week 0

    You are publishing to PyPI every few weeks. Users install your library the day you publish. If there are no docs, they cannot use it. Documentation is not a phase 6 activity — it ships incrementally with every version.

    Doc stack you will use

    MkDocs + Material theme — generates a static doc site from Markdown files. Hosted free on GitHub Pages. Updates automatically when you push. This is what FastAPI, Pydantic, and httpx use.

    mkdocstrings — reads your Python docstrings and generates API reference pages automatically. Every docstring you write in your source files becomes a searchable API page. This is why docstrings matter from week 1.

    mike — manages versioned docs. When you publish v0.1, v0.2, v0.3, users can read docs for the exact version they installed. The landing page shows the latest stable version.

Docstring: "Emitted when resolve_batch() detects a path collision. The winning patch is applied. Both patches are stored as dicts for serializability."

    agentstatelib/router/graph.py

    Use the parallel round-based version from the review session. Every public method needs a complete docstring.

    AgentGraph class docstring must explain the execution model clearly: "Round-based parallel agent execution. Each round: (1) collect all agents whose edges fire, (2) run them concurrently against the same state snapshot, (3) batch-resolve any path conflicts, (4) apply winning patches, (5) check invariants, (6) repeat. This guarantees all agents in a round see consistent state and conflicts are resolved explicitly."

    run() method docstring: explain Parameters, Returns, Raises. Note that start accepts both str and list[str] for parallel start.

    node() decorator docstring: include a full working example in the docstring that users can copy.

    edge() docstring: explain that multiple edges from the same source agent cause all reachable agents to run in the next round.

@'
import asyncio
from agentstatelib import SharedState, AgentGraph, StatePatch

graph = AgentGraph()

@graph.node('planner', context=['goal'])
async def planner(ctx):
    return StatePatch(
        agent_id='planner',
        target='facts.plan',
        value='done',
        reason='planned',
    )

@graph.node('researcher', context=['facts'])
async def researcher(ctx):
    return StatePatch(
        agent_id='researcher',
        target='artifacts.result',
        value={
            'produced_by': 'researcher',
            'artifact_type': 'summary',
            'content': 'found',
        },
        reason='researched',
    )

graph.edge('planner', 'researcher', lambda s: s.get('facts', {}).get('plan') == 'done')

async def main():
    state = SharedState(goal='test workflow')
    final = await graph.run(state, start='planner')
    assert final.facts.get('plan') == 'done'
    assert final.artifacts['result'].content == 'found'
    print('quickstart example passes')

asyncio.run(main())
'@ | python

    docs/concepts/parallelism.md

    Write the Parallel Execution concept guide. Sections:

    The round model — every round all runnable agents execute concurrently against the same state snapshot. No agent in a round sees another's updates until the next round.

    Why this matters — eliminates read-write races within a round. Contrast with sequential execution where agent B always sees agent A's changes.

    Fan-out topology — how coordinator→[analyst_a, analyst_b] works. Both analysts run in round 2 simultaneously.

    Fan-in — how [analyst_a, analyst_b]→aggregator works. Aggregator runs in round 3 after both analysts complete.

    Parallel start — run(state, start=["a", "b"]) for when multiple agents should start simultaneously.

    The semaphore — max_concurrent parameter in AgentGraph.__init__. Controls total concurrency. Default 10.

import asyncio
from agentstatelib import SharedState, AgentGraph, StatePatch
from agentstatelib.core.events import ConflictDetected
from agentstatelib.memory.store import InMemoryStore

async def main():
    store = InMemoryStore()
    graph = AgentGraph(store=store)

    @graph.node('analyst_a')
    async def analyst_a(ctx):
        return StatePatch(agent_id='analyst_a', target='facts.verdict', value='positive', reason='a says positive')

    @graph.node('analyst_b')
    async def analyst_b(ctx):
        return StatePatch(agent_id='analyst_b', target='facts.verdict', value='negative', reason='b says negative')

    state = SharedState(goal='conflict test')
    final = await graph.run(state, start=['analyst_a', 'analyst_b'])

    events = await store.get_workflow(state.workflow_id)
    conflicts = [e for e in events if isinstance(e, ConflictDetected)]
    print(f'Conflicts: {len(conflicts)}')
    print(f'Winner: {conflicts[0].winner_agent_id}')
    print(f'Final verdict: {final.facts['verdict']}')
    assert len(conflicts) == 1
    print('conflict test passed')

asyncio.run(main())
